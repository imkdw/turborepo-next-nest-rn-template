# /decompose: 문제 분해 및 실행 계획 생성 명령어

## 사용법
```
/decompose "복잡한 문제나 이슈를 여기에 입력"
```

## 명령어 동작 방식

복잡한 문제 "$ARGUMENT"를 체계적으로 분해하여 실행 가능한 작은 단위의 작업으로 나누고, 구조화된 실행 계획을 생성합니다.

### 1. 문제 분석 및 이해
**목표**: $ARGUMENT의 핵심 요구사항과 제약사항 파악
- 📋 **문제 정의**: 해결해야 할 핵심 문제가 무엇인지 명확히 정의
- 🎯 **최종 목표**: 달성하고자 하는 구체적인 결과 설정
- ⚖️ **제약사항**: 시간, 리소스, 기술적 한계 파악
- 📊 **성공 지표**: 문제 해결 완료를 판단할 기준 설정

### 2. 계층적 분해 구조
**목표**: 큰 문제를 관리 가능한 단위로 체계적 분해

#### 🏗️ Phase 분해 (상위 레벨)
- **Phase 1~N**: 논리적 순서와 의존성에 따른 주요 단계 구분
- **각 Phase별 목표**: 단계별 달성해야 할 구체적 결과물
- **Phase 간 의존성**: 순차적 vs 병렬 처리 가능성 분석

#### 📦 Task 분해 (중간 레벨)  
- **Task 세분화**: 각 Phase를 실행 가능한 작업 단위로 분할
- **예상 소요 시간**: 각 Task별 작업 시간 추정
- **필요 리소스**: 도구, 기술, 지식, 권한 요구사항

#### ✅ Todo 분해 (실행 레벨)
- **구체적 행동**: SMART 원칙에 따른 명확한 실행 항목
- **완료 조건**: 각 Todo의 성공/실패 판단 기준
- **검증 방법**: 작업 완료 후 품질 확인 방안

### 3. 의존성 및 우선순위 매핑
**목표**: 효율적 실행 순서와 병목점 파악
- 🔄 **의존성 체인**: 선행 작업이 필요한 Task 식별
- ⚡ **병렬 처리**: 동시 실행 가능한 작업 그룹핑
- 🚨 **중요도 분석**: High/Medium/Low 우선순위 설정
- ⏰ **크리티컬 패스**: 전체 일정에 영향을 주는 핵심 경로

### 4. 위험 요소 및 대안 계획
**목표**: 잠재적 문제점 예측 및 대응방안 수립
- ⚠️ **위험 요소**: 실행 과정에서 발생 가능한 문제점
- 📈 **영향도 평가**: 위험이 현실화될 경우의 파급효과
- 🛡️ **완화 전략**: 위험 발생 확률을 줄이는 예방책
- 🔄 **대안 경로**: Plan B, C 등 우회 방안

### 5. 검증 및 품질 관리
**목표**: 각 단계별 품질 보증과 최종 목표 달성 확인
- ✔️ **단계별 검증**: Phase/Task 완료 후 품질 체크포인트
- 📏 **품질 기준**: 각 단계에서 만족해야 할 최소 요구사항
- 🔍 **중간 점검**: 진행 상황 점검 및 방향 조정 포인트
- 🎯 **최종 검증**: 전체 문제 해결 완료 여부 확인

### 6. 리소스 및 일정 계획
**목표**: 현실적이고 실행 가능한 계획 수립
- 👥 **인력 계획**: 필요한 역할과 책임 분배
- 🛠️ **도구/기술**: 각 단계에서 필요한 기술 스택과 도구
- ⏱️ **타임라인**: 전체 일정과 마일스톤 설정
- 💰 **예산/비용**: 예상 소요 비용과 리소스 투입량

## 명령어 활용 예시

```bash
/decompose "웹사이트 성능을 개선하고 싶은데 로딩이 너무 느리고 모바일에서 사용하기 어려워"

/decompose "팀의 코드 리뷰 프로세스를 개선해서 품질을 높이고 개발 속도도 향상시키고 싶어"

/decompose "레거시 시스템을 마이크로서비스 아키텍처로 전환하면서 서비스 중단 없이 진행하려고 해"
```

## 주요 특징
- ✅ **체계적 접근**: 문제를 논리적 순서로 분해
- ✅ **실행 중심**: 추상적 계획이 아닌 구체적 행동 중심
- ✅ **위험 관리**: 잠재적 문제점과 대응방안 포함
- ✅ **품질 보증**: 각 단계별 검증 포인트 설정
- ✅ **유연성**: 진행 중 조정 가능한 구조
